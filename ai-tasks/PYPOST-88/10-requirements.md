# PYPOST-88: Requirements — No pytest cov-fail-under threshold

**Date**: 2026-03-25
**Source**: ai-tasks/PYPOST-52/60-review.md (finding TD-6, LOW severity)
**Prerequisite**: PYPOST-89 CI pipeline merged (commit `9f0a52d`)

---

## 1. Background

The CI workflow (`.github/workflows/test.yml`) runs `pytest` with `--cov=pypost`,
`--cov-report=term-missing`, and `--cov-report=xml:coverage.xml`, but **does not enforce a
minimum coverage percentage**. The `Makefile` `test-cov` target is in the same situation.
This means a refactoring regression that reduces coverage to 0% will not fail the CI build —
the coverage data is collected but never acted upon.

The original review (PYPOST-52 `60-review.md`) recommended adding `--cov-fail-under` with a
suggested floor of 70%.

---

## 2. Problem Statement

Without a coverage floor:

- A PR that deletes tests or removes coverage passes CI silently.
- Engineers receive no automated signal when coverage regresses during refactoring
  (PYPOST-43–51 series).
- Coverage XML/HTML reports generated by CI are informational-only, providing no enforcement.

---

## 3. Goals

1. Enforce a minimum line-coverage percentage for the `pypost` package on every CI run.
2. Fail the CI build (non-zero exit) if coverage drops below the threshold.
3. Keep `make test-cov` consistent with CI — the same threshold must apply locally.
4. Choose a threshold that is realistic given the current coverage baseline, provides
   meaningful protection, and does not immediately break existing builds.

---

## 4. Out of Scope

- Branch coverage enforcement (line coverage is sufficient for this debt item).
- Per-module or per-file coverage exclusion rules (beyond what already exists).
- Raising the threshold above the initial agreed value (can be a follow-up ticket).
- Adding new tests to raise coverage (separate effort).

---

## 5. Baseline Audit Requirement

Before committing to a threshold value, a baseline run **must** be performed:

```bash
make test-cov
```

The total coverage percentage printed in the terminal output (`TOTAL ... XX%`) is the
**current baseline**. The chosen threshold must be:

- **At or below** the current baseline (to avoid immediately breaking CI).
- **At least** `baseline - 5pp` floor, rounded down to the nearest 5 % increment, to give
  a small buffer for incidental fluctuation while still providing meaningful protection.

**Example**: if the baseline is 74%, the threshold should be set to 70%.

If the baseline is already at or above 70%, the threshold should be set to **70%** as the
agreed minimum for this project.

If the baseline is below 70%, the threshold should be set to `floor(baseline / 5) * 5`
(rounded down to nearest 5%), documented in this ticket, and a follow-up ticket created to
raise it to 70%.

---

## 6. Functional Requirements

### FR-1: CI enforcement

The `--cov-fail-under=<THRESHOLD>` flag **must** be added to the `pytest` invocation in
`.github/workflows/test.yml` (the `Run tests` step). The CI job must exit with a non-zero
status when coverage is below the threshold.

### FR-2: Makefile consistency

The `test-cov` target in `Makefile` **must** include the same `--cov-fail-under=<THRESHOLD>`
flag so local runs mirror CI behaviour.

### FR-3: Single source of truth (preferred)

The threshold **should** be defined in `pytest.ini` via the `addopts` option so that both
the Makefile and the GitHub Actions workflow inherit it automatically, without duplication:

```ini
[pytest]
addopts = -v --tb=short --cov-fail-under=70
```

If `pytest.ini` is used as the source, the explicit `--cov-fail-under` flag in `Makefile`
and `test.yml` can be omitted (as long as those invocations trigger pytest normally and pick
up `addopts`).

### FR-4: Threshold value

The agreed threshold is **70%** (subject to baseline audit per Section 5). This value must
be recorded in a comment wherever it is set so that future engineers know the rationale.

### FR-5: No new test files

This change must not add or modify any test file. It is a configuration-only change.

---

## 7. Non-Functional Requirements

- **NFR-1**: The change must be minimal — only the files that configure coverage threshold
  need to be modified (`pytest.ini`, `Makefile`, `.github/workflows/test.yml`).
- **NFR-2**: The CI pipeline must continue to pass on the current codebase after the change
  (i.e., the chosen threshold must be ≤ current baseline).
- **NFR-3**: The enforcement must be visible in CI output — the `--cov-fail-under` failure
  message from pytest-cov must appear in the job log when coverage is below threshold.

| # | Criterion | Verification |
